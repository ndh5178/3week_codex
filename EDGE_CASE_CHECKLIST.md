# Edge Case Checklist

이 문서는 Mini Redis Board 프로젝트에서 후속 검증이 필요한 엣지 케이스를 정리한 체크리스트다.  
각 항목은 `기대 결과`를 먼저 정하고, 실제 확인 후 `상태`를 갱신하는 방식으로 사용하는 것을 기준으로 작성했다.

상태 표기 예시:

- `PENDING`: 아직 확인하지 않음
- `PASS`: 기대 결과대로 동작 확인
- `FAIL`: 기대 결과와 다름

## 1. Mini Redis 기본 동작

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 없는 key에 `get` 요청 | `value=None` 또는 없는 값으로 처리 | PENDING |
| 없는 key에 `exists` 요청 | 존재하지 않는 것으로 반환 | PENDING |
| 없는 key에 `ttl` 요청 | 키 없음 상태로 반환 | PENDING |
| 없는 key에 `delete` 요청 | 삭제 수 0 또는 false 성격의 응답 | PENDING |
| `clear` 직후 `get` 요청 | 모든 key가 비어 있어야 함 | PENDING |

## 2. TTL / 만료 처리

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| `setex` 후 TTL 안에 `get` 요청 | 값 정상 반환 | PENDING |
| TTL 만료 직후 `get` 요청 | 값 제거 후 없는 값처럼 반환 | PENDING |
| TTL 만료 직후 `exists` 요청 | 존재하지 않는 것으로 반환 | PENDING |
| TTL 만료 직후 `ttl` 요청 | 만료 또는 키 없음 상태로 반환 | PENDING |
| TTL이 0 이하인 `setex` 요청 | 예외 또는 오류 응답 | PENDING |

## 3. 숫자 연산

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 없는 key에 `incr` 요청 | 1부터 시작해서 저장 | PENDING |
| 정수 값에 `incr` 요청 | 값이 1 증가 | PENDING |
| 문자열 값에 `incr` 요청 | 오류 반환 | PENDING |
| TTL이 있는 숫자 key에 `incr` 요청 | 만료 여부 처리 후 증가 결과 반환 | PENDING |

## 4. 게시글 캐시 흐름

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 첫 게시글 조회 | DB read 후 캐시 저장 | PENDING |
| 같은 게시글 재조회 | cache hit로 처리 | PENDING |
| 게시글 수정 후 재조회 | 기존 캐시 무효화 후 새 값 반영 | PENDING |
| 게시글 삭제 후 재조회 | 조회 실패 또는 404 | PENDING |
| 특정 게시글 캐시 수동 삭제 후 재조회 | 다시 DB에서 읽고 캐시 재생성 | PENDING |

## 5. 세션 처리

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 정상 로그인 후 세션 확인 | 인증 성공 | PENDING |
| 로그아웃 후 세션 확인 | 인증 실패 | PENDING |
| 만료된 세션으로 확인 요청 | 인증 실패 | PENDING |
| 빈 username 로그인 요청 | 오류 반환 | PENDING |

## 6. 인기글 캐시 / 조회수

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 첫 인기글 조회 | 계산 후 캐시 저장 | PENDING |
| 같은 조건으로 재조회 | cache hit로 처리 | PENDING |
| 조회수 증가 후 인기글 조회 | 기존 인기글 캐시 무효화 후 재계산 | PENDING |
| 여러 번 조회 시 조회수 증가 | 값이 누락 없이 누적 | PENDING |

## 7. 동시성 / 빠른 연속 요청

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 같은 게시글 빠른 연속 클릭 | 조회수 값이 크게 어긋나지 않아야 함 | PENDING |
| 동시에 여러 key 저장 | 메모리 상태가 깨지지 않아야 함 | PENDING |
| 만료와 조회가 겹치는 상황 | 만료된 값은 안전하게 제거되어야 함 | PENDING |

## 8. 복구 / JSON dump

| 항목 | 기대 결과 | 상태 |
| --- | --- | --- |
| 값 저장 후 dump 파일 생성 | `store`, `expire_at` 구조로 저장 | PENDING |
| 서버 재시작 후 dump 로드 | 기존 값이 메모리에 복구 | PENDING |
| 만료된 값이 dump에 남아 있는 경우 | 로드 후 정리 또는 조회 시 제거 | PENDING |
| `clear` 후 재시작 | 이전 값이 남아 있지 않아야 함 | PENDING |

## 9. 체크 순서 제안

1. Mini Redis 기본 동작
2. TTL / 만료 처리
3. 숫자 연산
4. 게시글 캐시 흐름
5. 세션 처리
6. 인기글 캐시 / 조회수
7. 복구 / JSON dump
8. 동시성 / 빠른 연속 요청

## 10. 기록 방법

- 실제 확인 날짜를 남긴다.
- `PASS` 또는 `FAIL`로 상태를 갱신한다.
- 실패한 항목은 재현 조건과 원인을 한 줄로 기록한다.
- 발표 자료나 README에는 `PASS` 된 항목만 근거로 적는다.
